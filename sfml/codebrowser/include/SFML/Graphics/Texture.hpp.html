416 ////////////////////////////////////////////////////////////
417 /// \brief Enable or disable conversion from sRGB
418 ///
419 /// When providing texture data from an image file or memory, it can
420 /// either be stored in a linear color space or an sRGB color space.
421 /// Most digital images account for gamma correction already, so they
422 /// would need to be "uncorrected" back to linear color space before
423 /// being processed by the hardware. The hardware can automatically
424 /// convert it from the sRGB color space to a linear color space when
425 /// it gets sampled. When the rendered image gets output to the final
426 /// framebuffer, it gets converted back to sRGB.
427 ///
428 /// After enabling or disabling sRGB conversion, make sure to reload
429 /// the texture data in order for the setting to take effect.
430 ///
431 /// This option is only useful in conjunction with an sRGB capable
432 /// framebuffer. This can be requested during window creation.
433 ///
434 /// \param sRgb True to enable sRGB conversion, false to disable it
435 ///
436 /// \see isSrgb
437 ///
438 ////////////////////////////////////////////////////////////
439 void setSrgb(bool sRgb);
440
441 ////////////////////////////////////////////////////////////
442 /// \brief Tell whether the texture source is converted from sRGB or not

444 /// \return True if the texture source is converted from sRGB, false if not
445 ///
446 /// \see setSrgb
447 ///
448 ////////////////////////////////////////////////////////////
449 bool isSrgb() const;

451 ////////////////////////////////////////////////////////////
452 /// \brief Enable or disable repeating
453 ///
454 /// Repeating is involved when using texture coordinates
455 /// outside the texture rectangle [0, 0, width, height].
456 /// In this case, if repeat mode is enabled, the whole texture
457 /// will be repeated as many times as needed to reach the
458 /// coordinate (for example, if the X texture coordinate is
459 /// 3 * width, the texture will be repeated 3 times).
460 /// If repeat mode is disabled, the "extra space" will instead
461 /// be filled with border pixels.
462 /// Warning: on very old graphics cards, white pixels may appear
463 /// when the texture is repeated. With such cards, repeat mode
464 /// can be used reliably only if the texture has power-of-two
465 /// dimensions (such as 256x128).
466 /// Repeating is disabled by default.
467 ///
468 /// \param repeated True to repeat the texture, false to disable repeating
469 ///
470 /// \see isRepeated
471 ///
472 ////////////////////////////////////////////////////////////
473 void setRepeated(bool repeated);

480 /// \see setRepeated

483 bool isRepeated() const;

485 ////////////////////////////////////////////////////////////
486 /// \brief Generate a mipmap using the current texture data
487 ///
488 /// Mipmaps are pre-computed chains of optimized textures. Each
489 /// level of texture in a mipmap is generated by halving each of
490 /// the previous level's dimensions. This is done until the final
491 /// level has the size of 1x1. The textures generated in this process may
492 /// make use of more advanced filters which might improve the visual quality
493 /// of textures when they are applied to objects much smaller than they are.
494 /// This is known as minification. Because fewer texels (texture elements)
495 /// have to be sampled from when heavily minified, usage of mipmaps
496 /// can also improve rendering performance in certain scenarios.
497 ///
498 /// Mipmap generation relies on the necessary OpenGL extension being
499 /// available. If it is unavailable or generation fails due to another
500 /// reason, this function will return false. Mipmap data is only valid from
501 /// the time it is generated until the next time the base level image is
502 /// modified, at which point this function will have to be called again to
503 /// regenerate it.
504 ///
505 /// \return True if mipmap generation was successful, false if unsuccessful
506 ///
507 ////////////////////////////////////////////////////////////
508 bool generateMipmap();

540 ////////////////////////////////////////////////////////////
541 /// \brief Bind a texture for rendering
542 ///
543 /// This function is not part of the graphics API, it mustn't be
544 /// used when drawing SFML entities. It must be used only if you
545 /// mix sf::Texture with OpenGL code.
546 ///
547 /// \code
548 /// sf::Texture t1, t2;
549 /// ...
550 /// sf::Texture::bind(&t1);
551 /// // draw OpenGL stuff that use t1...
552 /// sf::Texture::bind(&t2);
553 /// // draw OpenGL stuff that use t2...
554 /// sf::Texture::bind(NULL);
555 /// // draw OpenGL stuff that use no texture...
556 /// \endcode
557 ///
558 /// The \a coordinateType argument controls how texture
559 /// coordinates will be interpreted. If Normalized (the default), they
560 /// must be in range [0 .. 1], which is the default way of handling
561 /// texture coordinates with OpenGL. If Pixels, they must be given
562 /// in pixels (range [0 .. size]). This mode is used internally by
563 /// the graphics classes of SFML, it makes the definition of texture
564 /// coordinates more intuitive for the high-level API, users don't need
565 /// to compute normalized values.
566 ///
567 /// \param texture Pointer to the texture to bind, can be null to use no texture
568 /// \param coordinateType Type of texture coordinates to use
569 ///
570 ////////////////////////////////////////////////////////////
571 static void bind(const Texture* texture, CoordinateType coordinateType = Normalized);

634////////////////////////////////////////////////////////////
635/// \class sf::Texture
636/// \ingroup graphics
637///
638/// sf::Texture stores pixels that can be drawn, with a sprite
639/// for example. A texture lives in the graphics card memory,
640/// therefore it is very fast to draw a texture to a render target,
641/// or copy a render target to a texture (the graphics card can
642/// access both directly).
643///
644/// Being stored in the graphics card memory has some drawbacks.
645/// A texture cannot be manipulated as freely as a sf::Image,
646/// you need to prepare the pixels first and then upload them
647/// to the texture in a single operation (see Texture::update).
648///
649/// sf::Texture makes it easy to convert from/to sf::Image, but
650/// keep in mind that these calls require transfers between
651/// the graphics card and the central memory, therefore they are
652/// slow operations.
653///
654/// A texture can be loaded from an image, but also directly
655/// from a file/memory/stream. The necessary shortcuts are defined
656/// so that you don't need an image first for the most common cases.
657/// However, if you want to perform some modifications on the pixels
658/// before creating the final texture, you can load your file to a
659/// sf::Image, do whatever you need with the pixels, and then call
660/// Texture::loadFromImage.
661///
662/// Since they live in the graphics card memory, the pixels of a texture
663/// cannot be accessed without a slow copy first. And they cannot be
664/// accessed individually. Therefore, if you need to read the texture's
665/// pixels (like for pixel-perfect collisions), it is recommended to
666/// store the collision information separately, for example in an array
667/// of booleans.
668///
669/// Like sf::Image, sf::Texture can handle a unique internal
670/// representation of pixels, which is RGBA 32 bits. This means
671/// that a pixel must be composed of 8 bits red, green, blue and
672/// alpha channels -- just like a sf::Color.
673///
674/// Usage example:
675/// \code
676/// // This example shows the most common use of sf::Texture:
677/// // drawing a sprite
678///
679/// // Load a texture from a file
680/// sf::Texture texture;
681/// if (!texture.loadFromFile("texture.png"))
682/// return -1;
683///
684/// // Assign it to a sprite
685/// sf::Sprite sprite;
686/// sprite.setTexture(texture);
687///
688/// // Draw the textured sprite
689/// window.draw(sprite);
690/// \endcode
691///
692/// \code
693/// // This example shows another common use of sf::Texture:
694/// // streaming real-time data, like video frames
695///
696/// // Create an empty texture
697/// sf::Texture texture;
698/// if (!texture.create(640, 480))
699/// return -1;
700///
701/// // Create a sprite that will display the texture
702/// sf::Sprite sprite(texture);
703///
704/// while (...) // the main loop
705/// {
706/// ...
707///
708/// // update the texture
709/// sf::Uint8* pixels = ...; // get a fresh chunk of pixels (the next frame of a movie, for example)
710/// texture.update(pixels);
711///
712/// // draw it
713/// window.draw(sprite);
714///
715/// ...
716/// }
717///
718/// \endcode
719///
720/// Like sf::Shader that can be used as a raw OpenGL shader,
721/// sf::Texture can also be used directly as a raw texture for
722/// custom OpenGL geometry.
723/// \code
724/// sf::Texture::bind(&texture);
725/// ... render OpenGL geometry ...
726/// sf::Texture::bind(NULL);
727/// \endcode
728///
729/// \see sf::Sprite, sf::Image, sf::RenderTexture
730///
