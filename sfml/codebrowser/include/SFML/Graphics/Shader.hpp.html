101 /// \brief Load the vertex, geometry or fragment shader from a file
102 ///
103 /// This function loads a single shader, vertex, geometry or
104 /// fragment, identified by the second argument.
105 /// The source must be a text file containing a valid
106 /// shader in GLSL language. GLSL is a C-like language
107 /// dedicated to OpenGL shaders; you'll probably need to
108 /// read a good documentation for it before writing your
109 /// own shaders.
110 ///

433 ////////////////////////////////////////////////////////////
434 /// \brief Specify a texture as \p sampler2D uniform
435 ///
436 /// \a name is the name of the variable to change in the shader.
437 /// The corresponding parameter in the shader must be a 2D texture
438 /// (\p sampler2D GLSL type).
439 ///
440 /// Example:
441 /// \code
442 /// uniform sampler2D the_texture; // this is the variable in the shader
443 /// \endcode
444 /// \code
445 /// sf::Texture texture;
446 /// ...
447 /// shader.setUniform("the_texture", texture);
448 /// \endcode
449 /// It is important to note that \a texture must remain alive as long
450 /// as the shader uses it, no copy is made internally.
451 ///
452 /// To use the texture of the object being drawn, which cannot be
453 /// known in advance, you can pass the special value
454 /// sf::Shader::CurrentTexture:
455 /// \code
456 /// shader.setUniform("the_texture", sf::Shader::CurrentTexture).
457 /// \endcode
458 ///
459 /// \param name Name of the texture in the shader
460 /// \param texture Texture to assign
461 ///
462 ////////////////////////////////////////////////////////////
463 void setUniform(const std::string& name, const Texture& texture);
464
465 ////////////////////////////////////////////////////////////
466 /// \brief Specify current texture as \p sampler2D uniform
467 ///
468 /// This overload maps a shader texture variable to the
469 /// texture of the object being drawn, which cannot be

472 /// The corresponding parameter in the shader must be a 2D texture
473 /// (\p sampler2D GLSL type).
474 ///
475 /// Example:
476 /// \code
477 /// uniform sampler2D current; // this is the variable in the shader
478 /// \endcode
479 /// \code
480 /// shader.setUniform("current", sf::Shader::CurrentTexture);
481 /// \endcode
482 ///
483 /// \param name Name of the texture in the shader
484 ///
485 ////////////////////////////////////////////////////////////

761/// \class sf::Shader
762/// \ingroup graphics
763///
764/// Shaders are programs written using a specific language,
765/// executed directly by the graphics card and allowing
766/// to apply real-time operations to the rendered entities.
767///
768/// There are three kinds of shaders:
769/// \li %Vertex shaders, that process vertices
770/// \li Geometry shaders, that process primitives
771/// \li Fragment (pixel) shaders, that process pixels
772///
773/// A sf::Shader can be composed of either a vertex shader
774/// alone, a geometry shader alone, a fragment shader alone,
775/// or any combination of them. (see the variants of the
776/// load functions).
777///
778/// Shaders are written in GLSL, which is a C-like
779/// language dedicated to OpenGL shaders. You'll probably
780/// need to learn its basics before writing your own shaders
781/// for SFML.
782///
783/// Like any C/C++ program, a GLSL shader has its own variables
784/// called \a uniforms that you can set from your C++ application.
785/// sf::Shader handles different types of uniforms:
786/// \li scalars: \p float, \p int, \p bool
787/// \li vectors (2, 3 or 4 components)
788/// \li matrices (3x3 or 4x4)
789/// \li samplers (textures)
790///
791/// Some SFML-specific types can be converted:
792/// \li sf::Color as a 4D vector (\p vec4)
793/// \li sf::Transform as matrices (\p mat3 or \p mat4)
794///
795/// Every uniform variable in a shader can be set through one of the
796/// setUniform() or setUniformArray() overloads. For example, if you
797/// have a shader with the following uniforms:
798/// \code
799/// uniform float offset;
800/// uniform vec3 point;
801/// uniform vec4 color;
802/// uniform mat4 matrix;
803/// uniform sampler2D overlay;
804/// uniform sampler2D current;
805/// \endcode
806/// You can set their values from C++ code as follows, using the types
807/// defined in the sf::Glsl namespace:
808/// \code
809/// shader.setUniform("offset", 2.f);
810/// shader.setUniform("point", sf::Vector3f(0.5f, 0.8f, 0.3f));
811/// shader.setUniform("color", sf::Glsl::Vec4(color)); // color is a sf::Color
812/// shader.setUniform("matrix", sf::Glsl::Mat4(transform)); // transform is a sf::Transform
813/// shader.setUniform("overlay", texture); // texture is a sf::Texture
814/// shader.setUniform("current", sf::Shader::CurrentTexture);
815/// \endcode
816///
817/// The old setParameter() overloads are deprecated and will be removed in a
818/// future version. You should use their setUniform() equivalents instead.
819///
820/// The special Shader::CurrentTexture argument maps the
821/// given \p sampler2D uniform to the current texture of the
822/// object being drawn (which cannot be known in advance).
823///
824/// To apply a shader to a drawable, you must pass it as an
825/// additional parameter to the \ref Window::draw() draw() function:
826/// \code
827/// window.draw(sprite, &shader);
828/// \endcode
829///
830/// ... which is in fact just a shortcut for this:
831/// \code
832/// sf::RenderStates states;
833/// states.shader = &shader;
834/// window.draw(sprite, states);
835/// \endcode
836///
837/// In the code above we pass a pointer to the shader, because it may
838/// be null (which means "no shader").
839///
840/// Shaders can be used on any drawable, but some combinations are
841/// not interesting. For example, using a vertex shader on a sf::Sprite
842/// is limited because there are only 4 vertices, the sprite would
843/// have to be subdivided in order to apply wave effects.
844/// Another bad example is a fragment shader with sf::Text: the texture
845/// of the text is not the actual text that you see on screen, it is
846/// a big texture containing all the characters of the font in an
847/// arbitrary order; thus, texture lookups on pixels other than the
848/// current one may not give you the expected result.
849///
850/// Shaders can also be used to apply global post-effects to the
851/// current contents of the target (like the old sf::PostFx class
852/// in SFML 1). This can be done in two different ways:
853/// \li draw everything to a sf::RenderTexture, then draw it to
854/// the main target using the shader
855/// \li draw everything directly to the main target, then use
856/// sf::Texture::update(Window&) to copy its contents to a texture
857/// and draw it to the main target using the shader
858///
859/// The first technique is more optimized because it doesn't involve
860/// retrieving the target's pixels to system memory, but the
861/// second one doesn't impact the rendering process and can be
862/// easily inserted anywhere without impacting all the code.
863///
864/// Like sf::Texture that can be used as a raw OpenGL texture,
865/// sf::Shader can also be used directly as a raw shader for
866/// custom OpenGL geometry.
867/// \code
868/// sf::Shader::bind(&shader);
869/// ... render OpenGL geometry ...
870/// sf::Shader::bind(NULL);
871/// \endcode
872///
873/// \see sf::Glsl
874///
